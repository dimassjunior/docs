I'm **Dimas Jr.** — a software developer, tinkerer, and builder. I got my start digging into the internals of games and gradually expanded outward into systems programming, web development, and open-source collaboration. I believe the best software is built by people who genuinely love what they make, and that ethos runs through everything I ship.

My work spans low-level C\+\+ engine code and high-level TypeScript interfaces — sometimes in the same afternoon. I care deeply about the developer experience on both sides of the equation: writing code that's a pleasure to maintain, and building products that feel effortless to use.

## What I Build

### Systems & Game Programming

My roots are in the **MTA:San Andreas** modding scene, where I learned to navigate complex C\+\+ codebases, hook into engine internals, and push a decade-old game engine well beyond its original design. This work led to [Midnight Purple SA](/midnight-purple/introduction) — a full fork of MTA:SA that integrates the **FMOD Core SDK** for advanced audio capabilities.

- **C\+\+** for engine-level modifications, audio pipeline integration, and native performance
- **Lua** for game scripting, server logic, and exposing engine features to mod developers
- Deep experience with **GTA:SA / MTA:SA** internals, DirectX, and Win32 APIs

### Modern Web Development

On the web side, I build fast, maintainable, and visually polished frontends. My current stack centres on:

- **React 18** with functional components and modern hooks patterns
- **TypeScript** for type-safe, refactorable codebases
- **Vite** for blazing-fast development and optimised production builds
- **React Router** for client-side navigation and nested route architectures
- **i18next** for full internationalisation (PT-BR and EN) with runtime language switching
- **Anime.js v4** for fluid, declarative UI animations
- **CSS Modules** for scoped, conflict-free styling

### Open Source

I'm an advocate for open, collaborative software. I build in the open when I can, document thoroughly, and try to leave every project better than I found it. Contributing to and maintaining public repositories is a core part of how I work and learn.

## Philosophy

I believe in **modular systems** — software that can be understood in pieces, extended without fear, and replaced without drama. This means clear separation of concerns, explicit interfaces, and minimal hidden coupling.

I value **clean APIs** above all else. Whether I'm writing a Lua binding for C\+\+ audio code or designing a React component's props interface, the contract between pieces of a system should be obvious, consistent, and well-documented.

Above all, I optimise for **developer experience** — both for myself and for anyone else who might work with my code. Good documentation, sensible defaults, and thoughtful error messages are not afterthoughts; they're part of the product.
